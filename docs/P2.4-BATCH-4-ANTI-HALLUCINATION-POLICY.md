# P2.4 Batch 4 - Anti-Hallucination Policy and Review Loops

Date: 2026-04-08
Scope: policy definition for Paperclip-managed hierarchy on top of OpenClaw-hosted inference routing.

## B4.1 Policy scope and failure taxonomy

This policy applies to all Foreman v2 Paperclip agents that produce:

- technical recommendations
- operational run decisions
- customer-facing claims

Failure taxonomy:

1. **Unsupported claim**: assertion without source/evidence.
2. **Mismatched evidence**: cited artifact does not support the claim.
3. **Stale evidence**: claim references superseded state/config.
4. **Unsafe action suggestion**: recommendation conflicts with phase boundaries or safety rules.
5. **Silent uncertainty**: uncertain output presented as high-confidence fact.

## B4.2 Evidence-first response invariant

Before any critical claim, the agent must provide:

- the evidence source (`file path`, `run id`, `API result`, or reproducible command output)
- a confidence statement (`high`, `medium`, `low`)
- a clear unknowns list when evidence is partial

Blocked-response rule:

- If evidence is missing for a critical claim, the response must explicitly stop and request or gather evidence first.

## B4.3 Reviewer-loop trigger and closure protocol

Trigger conditions for mandatory reviewer loop:

- change affects shell/provisioning/runtime scripts
- change affects cloud spend/infra topology assumptions
- change modifies public-facing attribution or third-party claims
- change risks phase scope creep

Closure protocol:

1. Run `script-auditor`, `cost-infra-reviewer`, `license-reviewer`, `phase-boundary-keeper`.
2. Classify outcomes as `signed off`, `pass with notes`, or `findings`.
3. Resolve findings or log explicit acceptance with rationale.
4. Record gate outcomes in `docs/PHASE-1-PLAN.md`.

## B4.5 Human-in-the-loop trigger matrix

| Trigger | Required human approval? | Reason |
|---|---|---|
| New cloud resource provisioning or SKU change | Yes | Cost and blast radius |
| License/legal policy override | Yes | Compliance risk |
| Phase boundary exception (pulling future-scope into current batch) | Yes | Scope governance |
| Destructive production action | Yes | Irreversible impact |
| Pure documentation sync with no behavioral change | No | Low risk |
